4. Technical Requirements & Challenges (from your comments on IPS-3867)
Client-side rendering (CSR) — current tool only handles server-side rendered pages. Needs headless browser support (Ondřej can do it but it's slower)
Facebook crawling — FB pages contain the exact company data needed; explore HTML scraping or API
AI/ML hardware — deep learning, LLM, RAG cannot run on current HW. Need GPU-capable hardware
Bloom filters — used for address extraction (removing non-address text)
Address validation via REST API — https://fcr-media.atlassian.net/browse/IPS-3842
Data linking — scraped pages ↔ uni_listing_id mapping is not 1:1, needs heuristics
KVK number confusion — regex for BE phones also picks up KVK numbers (you flagged ~1000 affected listings)
CommonCrawl.org — potential source for discovering additional company websites
Time-change comparison — track what appears/disappears on pages over time

5. Anti-Scraping (Our Side — Protecting FCR Data)
Multiple tickets focus on protecting FCR's own data. The new app should be aware of these patterns:
Block AI bots (ChatGPT, etc.) via robots.txt — https://fcr-media.atlassian.net/browse/IYP-1291
Anti-scraping proxy with IP blacklisting — https://fcr-media.atlassian.net/browse/IPS-3062, https://fcr-media.atlassian.net/browse/IR-115
Hide phone numbers in result lists — https://fcr-media.atlassian.net/browse/IYP-4235
Pagination limits (max 3 pages) — https://fcr-media.atlassian.net/browse/IPS-3576
Dummy/honeypot records for scraping detection — https://fcr-media.atlassian.net/browse/DT-6446

6. Key Architectural Ideas (PoV from DO-20)
https://fcr-media.atlassian.net/browse/DO-20 laid out the original PoV requirements that remain relevant:
Select/deselect different sources (Google, KBO, Infobel, etc.)
Select/deselect different fields per scrape
95% data quality target on results
Performant — full run within 1 month
Anonymous scraping (shouldn't cause issues on target sites)
Easy to add extra fields
CSV output + flexible reports
Manual validation of test results

7. Current Pipeline (from IPS-3867 comments)
Your current workflow as described in comments:
Ondřej receives URL file → runs crawl → stores HTML in zipped files on FTP (~monthly)
You retrieve data from FTP → run extraction tasks (regex for phones/emails, Bloom filters for addresses)
Results stored in structured files (CSV/JSON), keyed by URL hash
Map URLs → uni_listing_id → add to IPS

Bottom line: There's a rich history of scraping needs across the organization, from data enrichment to competitive intelligence to sales prospecting. The new app should consolidate all of this into a single platform that supports configurable extraction targets, handles modern CSR pages, integrates with IPS via the SCRAPING source, and runs validation workflows with the DQ team.

## basic requirements for web scraping app UI

Here's what I found across all tickets and meeting notes about UI requirements and use cases for the web scraping app:
UI Requirements & Use Cases Found
1. Existing "Webgrader Admin UI" (already built by Ondřej)
From https://fcr-media.atlassian.net/browse/DT-2767 and the https://fcr-media.atlassian.net/wiki/spaces/FCRTECH/pages/1240891402/Meeting+minutes+demo+crawler+by+Ondrej, there's already an admin UI at admin.webgrader.dev.fcr.local. It exposes:
Batch management — view batches of crawled URLs (e.g. /batches/119)
URL file management — upload URL selection files (e.g. ips_url_selection_be.csv) referenced in Ondřej's https://fcr-media.atlassian.net/browse/IPS-3867?focusedCommentId=134991
API on the crawler — programmatic access to trigger crawls, get screenshots, retrieve HTML content

From https://fcr-media.atlassian.net/browse/IPS-4247, two distinct UI use cases:
